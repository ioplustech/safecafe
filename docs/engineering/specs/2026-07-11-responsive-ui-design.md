# Safecafe 响应式 UI 修复设计

## 目标

修复 UI Sweep 与补充的多分辨率诊断中确认的四类问题，同时让测试能够持续发现断点附近的回归：

1. 768–820px 下 Agent 弹窗横向溢出。
2. 821–1181px 下验证者列表右侧状态与操作被裁切。
3. 320–360px 下验证者卡片和 Dashboard 持仓行字段重叠。
4. 移动端 Agent 浮球遮挡页面数据。

不调整桌面端品牌、导航信息架构、业务流程或链上交互逻辑；不把核心功能从移动端隐藏。

## 方案

### 1. Agent 响应式边界

- React 与 CSS 统一使用 `max-width: 820px` 作为移动 Agent 边界。
- `AgentLauncher` 使用 `matchMedia("(max-width: 820px)")` 跟踪窗口和方向变化。
- 在该范围内不向 `AgentChatDialog` 传入桌面锚点，使弹窗稳定使用全宽底部 Sheet。
- 自动断言 768px、820px 下弹窗四边均位于 viewport 内。

### 2. 中小屏 Agent 入口

- 在 `max-width: 1180px` 范围内，将圆形浮球改为贴在 viewport 右边缘的 `44×44px` Agent 标签。
- 即使启用 `VITE_AGENT_LAUNCHER_DRAGGABLE`，该范围也强制使用 compact/fixed 入口；拖拽仅保留在更宽的桌面布局。
- 使用左侧圆角、清晰图标、完整 `aria-label`，保持触控面积和可发现性。
- 标签固定在右下安全区；在 `max-width: 820px` 的移动布局中移除当前额外的 `72px` 上移量，并为页面底部保留安全留白。
- 不允许标签遮挡输入框、按钮等可交互元素；普通内容轻微相交由截图人工复核。

### 3. 验证者列表的内容驱动布局

- 为 `.validators-panel` 建立 inline-size container。
- 容器宽度不足以容纳 1000px 表格时，自动切换为卡片布局，而不是依赖 viewport 宽度。
- 中等宽度使用现有双列指标卡布局；桌面容器足够宽时保留表格。
- 这样可覆盖有侧栏的 821–1181px，以及未来不同父容器宽度。

### 4. 320–360px 窄屏布局

- 在 `max-width: 420px` 下，将验证者卡片改为单列区域：身份、状态、各项指标、操作依次排列。
- 操作区保持两个主要按钮与 44px 详情按钮可点击，不允许文字覆盖。
- Dashboard 的 `.position-row` 改为两列结构，金额移动到内容列下一行，进度条仍占满整行。
- 保留现有信息，不通过隐藏字段规避布局问题。

## UI Sweep 扩展

- 保留现有 1366×900 桌面完整流程和 390×844 移动完整流程。
- 每次测试额外执行一次响应式矩阵，覆盖：320、360、768、820、821、900、1024、1180、1181、1440、1920、2560px。
- 响应式矩阵至少截图 Dashboard、Validators、Settings 和 Agent。
- 自动检查：document 横向溢出、viewport 外的交互元素、Agent 弹窗越界、Agent 标签遮挡交互元素。
- 响应式矩阵只运行一次，不跟随三轮交互流程重复，控制执行时间和截图数量。

## 验证标准

- 所有目标分辨率无 document 横向溢出。
- 所有可交互元素完全位于 viewport 或明确的内部滚动容器中。
- 768px、820px Agent 底部 Sheet 无任何方向溢出。
- 320px、360px 验证者字段、状态和操作无重叠。
- 821–1181px 验证者状态和操作无需横向滚动即可访问。
- `pnpm check`、`pnpm test:e2e:ui-sweep`、`pnpm test:e2e`、`pnpm test:agent` 通过。

## 风险与边界

- Container Query 依赖现代浏览器；项目当前 Chromium E2E 与目标现代钱包浏览器均支持。
- 浏览器模拟不能替代真实 iOS Safari 与 Android Chrome，最终发布前仍建议各做一次真机抽查。
- 2560px 继续使用现有内容最大宽度，不为大屏无限拉伸数据面板。
